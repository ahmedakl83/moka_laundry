# وثيقة متطلبات المنتج (PRD) - تطبيق Moka Car Wash

**المؤلف:** Manus AI
**التاريخ:** 4 يونيو 2026
**الإصدار:** 1.0

## 1. المقدمة

تهدف هذه الوثيقة إلى تحديد المتطلبات الوظيفية وغير الوظيفية لتطبيق "مغسلة Moka"، وهو تطبيق محمول مخصص لنظام أندرويد يهدف إلى تبسيط وتسجيل العمليات اليومية للمغسلة. سيركز التطبيق على توفير واجهة سهلة الاستخدام للمدير والموظفين لتسجيل طلبات العملاء، تتبع المدفوعات، إدارة المصروفات، وتوفير تقارير شاملة لمراقبة الأداء. التطبيق مخصص للاستخدام الداخلي للمغسلة ولن يتم توزيعه تجاريًا عبر متاجر التطبيقات العامة.

## 2. الأهداف

*   تبسيط عملية تسجيل طلبات العملاء والخدمات المقدمة.
*   تتبع دقيق للمدفوعات والديون المستحقة على العملاء.
*   إدارة فعالة للمصروفات اليومية للمغسلة.
*   توفير تقارير شاملة لمساعدة المدير في اتخاذ القرارات.
*   ضمان أمان البيانات وسهولة الوصول إليها من قبل المستخدمين المصرح لهم.
*   توفير تجربة مستخدم سلسة وبسيطة.

## 3. الجمهور المستهدف

*   **المدير (Admin):** المسؤول عن الإعدادات العامة للتطبيق، إدارة المستخدمين، العملاء، الخدمات، فئات المصروفات، ومراجعة التقارير.
*   **موظف إدخال البيانات (Data Entry User):** المسؤول عن تسجيل طلبات العملاء، المدفوعات، والمصروفات اليومية.

## 4. المتطلبات الوظيفية

### 4.1. إدارة المدير (Admin Management)

| الميزة | الوصف | ملاحظات |
| :------ | :------ | :------ |
| **تسجيل الدخول الأولي** | عند تشغيل التطبيق لأول مرة، يتم استخدام بيانات اعتماد افتراضية: اسم المستخدم `admin` وكلمة المرور `admin123`. | يجب أن تكون هذه الخطوة لمرة واحدة فقط. |
| **تغيير بيانات الاعتماد الأولية** | بعد تسجيل الدخول الأولي، يجب على المدير تغيير اسم المستخدم وكلمة المرور الافتراضيين. | لتعزيز الأمان. |
| **إدخال معلومات المدير** | يجب على المدير إدخال اسمه الحقيقي وبريد جوجل الإلكتروني. | سيتم استخدام بريد جوجل لإثبات ملكية حساب المدير ولتخزين البيانات احتياطيًا على Google Drive. |
| **إدارة مستخدمي إدخال البيانات** | يمكن للمدير إضافة، تعديل، وحذف مستخدمي إدخال البيانات. لكل مستخدم: اسم حقيقي، اسم مستخدم، كلمة مرور، وبريد إلكتروني (جوجل). | يمكن استخدام بريد جوجل لربط المستخدم بحساباته الخاصة أو لأغراض استعادة كلمة المرور. |
| **إدارة قائمة العملاء** | يمكن للمدير إضافة، تعديل، وحذف العملاء. لكل عميل: اسم العميل ورقم هاتفه. | سيتم استخدام هذه القائمة عند تسجيل طلبات العملاء. |
| **إدارة أنواع الخدمات** | يمكن للمدير إضافة، تعديل، وحذف أنواع الخدمات التي تقدمها المغسلة. لكل خدمة: اسم الخدمة وسعرها. | سيتم استخدام هذه القائمة عند تسجيل طلبات العملاء. |
| **إدارة فئات المصروفات** | يمكن للمدير إضافة، تعديل، وحذف فئات المصروفات (مثل: إيجار، كهرباء، مواد تنظيف). | سيتم استخدام هذه القائمة عند تسجيل المصروفات اليومية. |

### 4.2. تسجيل دفتر اليومية (Data Entry)

| الميزة | الوصف | ملاحظات |
| :------ | :------ | :------ |
| **تسجيل طلب جديد** | واجهة لإدخال تفاصيل طلب العميل. | يجب أن تكون سهلة وسريعة الاستخدام. |
| **الرقم المسلسل** | يتم إنشاء رقم مسلسل فريد لكل طلب تلقائيًا. | يمكن أن يكون تسلسليًا أو يعتمد على التاريخ والوقت. |
| **اسم العميل** | حقل اختياري. يمكن الاختيار من قائمة العملاء التي أعدها المدير. إذا لم يتم الاختيار، يكون العميل الافتراضي هو "عميل نقدي". | يجب أن تكون القائمة قابلة للبحث. |
| **رقم السيارة** | حقل لإدخال رقم السيارة. | للمساعدة في تحديد الطلب. |
| **نوع الخدمة** | يمكن اختيار خدمة واحدة أو أكثر من قائمة الخدمات التي أعدها المدير. | يجب أن يتم احتساب السعر الإجمالي تلقائيًا بناءً على الخدمات المختارة. |
| **الحساب الإجمالي** | يتم احتسابه تلقائيًا بناءً على الخدمات المختارة. | يمكن تعديله يدويًا إذا لزم الأمر (مع تسجيل التعديل). |
| **تأكيد الدفع** | خيارات للدفع: مدفوع بالكامل، مدفوع جزئيًا، دفع لاحقًا. | إذا كان العميل مسجلاً (ليس "عميل نقدي")، يمكنه الدفع لاحقًا أو جزئيًا، ويتم احتساب المتبقي كدين عليه. |
| **تسجيل السداد** | آلية لتسجيل سداد الديون المستحقة على العملاء المسجلين. | يجب أن يظهر الرصيد المتبقي للعميل. |
| **الملاحظات** | حقل نصي لإضافة أي ملاحظات خاصة بالطلب. | |
| **ترحيل النقدية** | القائم بإتمام العملية (موظف إدخال البيانات) يحتسب أنه قام بقبض الحساب ويظل في ذمته حتى يتم ترحيل النقدية إلى المدير في نهاية اليوم. | يجب أن يكون هناك سجل واضح لعمليات الترحيل. |

### 4.3. المصروفات

| الميزة | الوصف | ملاحظات |
| :------ | :------ | :------ |
| **إدراج المصروفات اليومية** | واجهة لإدخال المصروفات اليومية. | يجب اختيار فئة المصروف من قائمة الفئات التي أعدها المدير. |
| **تفاصيل المصروف** | حقل لوصف المصروف والمبلغ. | |

### 4.4. التقارير (للمدير فقط)

| الميزة | الوصف | ملاحظات |
| :------ | :------ | :------ |
| **تقرير الإيرادات** | عرض الإيرادات حسب الفترة (يومي، أسبوعي، شهري)، حسب الخدمة، وحسب العميل. | يمكن أن يشمل الرسوم البيانية. |
| **تقرير المصروفات** | عرض المصروفات حسب الفترة (يومي، أسبوعي، شهري) وحسب الفئة. | يمكن أن يشمل الرسوم البيانية. |
| **تقرير الأرباح والخسائر** | ملخص للإيرادات والمصروفات لتحديد صافي الربح. | |
| **تقرير الديون المستحقة** | قائمة بالعملاء الذين عليهم ديون، مع تفاصيل المبالغ المستحقة. | |
| **تقرير أداء الموظفين** | تتبع أداء موظفي إدخال البيانات (عدد الطلبات، إجمالي المبالغ التي تم تحصيلها). | |

## 5. المتطلبات غير الوظيفية

### 5.1. الأداء

*   يجب أن يكون التطبيق سريع الاستجابة وخفيفًا على موارد الجهاز.
*   يجب أن يتم تحميل البيانات وعرضها بسرعة.

### 5.2. الأمان

*   حماية بيانات اعتماد المستخدمين (التشفير). [1]
*   التحقق من صحة المدخلات لمنع الثغرات الأمنية.
*   الوصول إلى البيانات بناءً على الأدوار (مدير/موظف).
*   تخزين البيانات الحساسة بشكل آمن.

### 5.3. سهولة الاستخدام (UI/UX)

*   واجهة مستخدم بسيطة، بديهية، وسهلة التنقل.
*   تصميم احترافي وجذاب مع استخدام اللون الأزرق كلون غالب.
*   سلاسة في إدخال البيانات والعمليات.
*   توفير رسائل تأكيد واضحة للمستخدم.

### 5.4. قابلية التوسع والصيانة

*   يجب أن يكون الكود منظمًا جيدًا وقابلاً للصيانة والتعديل في المستقبل.
*   يجب أن يكون التطبيق قادرًا على التعامل مع زيادة في حجم البيانات والعمليات.

### 5.5. التخزين والنسخ الاحتياطي

*   **التخزين المحلي:** يجب أن يخزن التطبيق البيانات محليًا لضمان العمل دون اتصال بالإنترنت ولتحسين الأداء.
*   **النسخ الاحتياطي على Google Drive:** يجب أن يقوم التطبيق بعمل نسخ احتياطية للبيانات على Google Drive في مجلد `appDataFolder` الخاص بالتطبيق، والذي لا يمكن للمستخدمين الوصول إليه مباشرة عبر واجهة Google Drive. [2]
    *   يجب أن يتم ربط النسخ الاحتياطي بحساب جوجل الخاص بالمدير لإثبات الملكية.
    *   يجب أن تكون هناك آلية للنسخ الاحتياطي التلقائي (مثلاً يوميًا) وخيار للنسخ الاحتياطي اليدوي.
    *   يجب أن تكون هناك آلية لاستعادة البيانات من Google Drive.

### 5.6. التوزيع

*   سيتم توزيع التطبيق مباشرة (APK) ولن يتم نشره على Google Play Store.

## 6. التقنيات المقترحة

بناءً على المتطلبات المذكورة، خاصة الحاجة إلى تطبيق خفيف الوزن، أندرويد فقط، وتجربة مستخدم سلسة، نقترح استخدام التقنيات التالية:

| الفئة | التقنية المقترحة | المبررات |
| :------ | :------ | :------ |
| **تطوير الواجهة الأمامية (Frontend)** | **Flutter (Dart)** | يوفر Flutter أداءً قريبًا من التطبيقات الأصلية، وواجهة مستخدم جميلة وقابلة للتخصيص بسهولة، ويدعم تطوير تطبيقات أندرويد بكفاءة. كما أنه يعتبر خفيف الوزن ويوفر تجربة تطوير سريعة. |
| **إدارة الحالة (State Management)** | **Provider / Riverpod** | حلول فعالة وخفيفة الوزن لإدارة حالة التطبيق في Flutter، مما يضمن أداءً جيدًا وسهولة في الصيانة. |
| **قاعدة البيانات المحلية (Local Database)** | **SQLite (عبر `sqflite` في Flutter)** | قاعدة بيانات علائقية خفيفة الوزن وموثوقة، مثالية لتخزين البيانات محليًا على الجهاز لضمان العمل دون اتصال بالإنترنت وتحسين سرعة الوصول للبيانات. |
| **التخزين السحابي والنسخ الاحتياطي** | **Google Drive API (App Data Folder)** | يسمح بتخزين البيانات الخاصة بالتطبيق بشكل آمن ومخفي عن المستخدمين، مما يضمن ملكية المدير للبيانات ويوفر آلية قوية للنسخ الاحتياطي والاستعادة. [2]
| **المصادقة (Authentication)** | **Firebase Authentication (اختياري)** | يمكن استخدامه لإدارة مصادقة المستخدمين (المدير وموظفي إدخال البيانات) بشكل آمن وفعال، مع دعم تسجيل الدخول باستخدام حسابات جوجل. |

## 7. خارطة طريق تقريبية (Roadmap)

1.  **المرحلة الأولى: الإعداد الأساسي**
    *   إعداد بيئة التطوير (Flutter).
    *   تطبيق شاشة تسجيل الدخول الأولية للمدير.
    *   تطبيق شاشة تغيير بيانات الاعتماد الأولية وإدخال معلومات المدير.
    *   إعداد التكامل الأساسي مع Google Drive API لتخزين بيانات المدير.
2.  **المرحلة الثانية: إدارة البيانات الأساسية**
    *   تطوير واجهات إدارة مستخدمي إدخال البيانات.
    *   تطوير واجهات إدارة قائمة العملاء.
    *   تطوير واجهات إدارة أنواع الخدمات وفئات المصروفات.
    *   تكامل قاعدة البيانات المحلية (SQLite).
3.  **المرحلة الثالثة: العمليات اليومية**
    *   تطوير واجهة تسجيل طلبات العملاء (دفتر اليومية).
    *   تطبيق منطق احتساب الأسعار وتتبع الدفعات والديون.
    *   تطوير واجهة تسجيل المصروفات اليومية.
    *   تطبيق آلية ترحيل النقدية.
4.  **المرحلة الرابعة: التقارير والتحسينات**
    *   تطوير شاشات التقارير للمدير (الإيرادات، المصروفات، الأرباح، الديون، أداء الموظفين).
    *   تحسينات UI/UX وتطبيق السمة اللونية الزرقاء.
    *   تطبيق النسخ الاحتياطي التلقائي والاستعادة من Google Drive.
    *   اختبار شامل للتطبيق وإصلاح الأخطاء.
